## Quiz – Intégration Continue & Déploiement Continu (CI/CD) – Version 2025



### Q1 — Objectif de l’intégration continue (CI)

**Question :**
Quel est l’objectif principal de l’intégration continue (CI) dans un projet logiciel moderne ?

* [ ] A. Permettre de livrer uniquement une fois par an.
* [ ] B. Automatiser le déploiement en production sans aucun test.
* [ ] C. Détecter les problèmes le plus tôt possible en intégrant fréquemment le code sur une branche commune avec build + tests automatiques.
* [ ] D. Remplacer les développeurs par des scripts.

<details>
  <summary>Réponse :</summary>

Bonne réponse : **C**.
La CI vise à intégrer fréquemment les changements dans une branche partagée et à exécuter automatiquement build + tests pour détecter les erreurs très tôt.

</details>

### Q2 — Continuous Delivery vs Continuous Deployment

**Question :**
Cochez la ou les affirmations correctes concernant la **Livraison Continue (Continuous Delivery)** et le **Déploiement Continu (Continuous Deployment)**.

* [ ] A. Continuous Delivery signifie que l’application est toujours dans un état déployable, mais le passage en production est encore une décision humaine.
* [ ] B. Continuous Deployment signifie que chaque changement validé est automatiquement déployé en production sans intervention humaine.
* [ ] C. Continuous Delivery et Continuous Deployment sont exactement la même chose.
* [ ] D. Continuous Delivery exclut l’automatisation des tests.

<details>
  <summary>Réponse :</summary>

Bonnes réponses : **A** et **B**.

* Continuous Delivery : l’application est toujours “prête à déployer”, mais la mise en prod reste une **décision humaine**.
* Continuous Deployment : la mise en prod est **automatique** dès que toutes les étapes du pipeline sont passées.

</details>

### Q3 — Rôle de Git dans un pipeline CI/CD

**Question :**
Pourquoi Git (et les plateformes comme GitHub/GitLab/Azure DevOps) est-il central dans un pipeline CI/CD moderne ?

* [ ] A. Parce qu’il remplace totalement les serveurs CI comme Jenkins.
* [ ] B. Parce qu’il fournit un dépôt central et des événements (push, pull request, tag) capables de déclencher automatiquement des pipelines CI/CD.
* [ ] C. Parce qu’il permet d’écrire du code en plusieurs langages.
* [ ] D. Parce qu’il remplace tous les outils de build (Maven, npm, etc.).

<details>
  <summary>Réponse :</summary>

Bonne réponse : **B**.
Git fournit un dépôt central avec historique, branches, pull requests, et génère des événements (webhooks, triggers) utilisés pour lancer automatiquement les pipelines CI/CD.

</details>

### Q4 — Pipeline as Code

**Question :**
Qu’entend-on par “Pipeline as Code” dans le contexte CI/CD ?

* [ ] A. Le pipeline est décrit dans un document Word validé par le chef de projet.
* [ ] B. Le pipeline (stages, jobs, tests, déploiements) est défini dans un fichier texte versionné dans Git (Jenkinsfile, `.github/workflows/*.yml`, `.gitlab-ci.yml`, etc.).
* [ ] C. Les développeurs ne sont plus autorisés à modifier le pipeline.
* [ ] D. Le pipeline est uniquement configuré via une interface graphique.

<details>
  <summary>Réponse :</summary>

Bonne réponse : **B**.
Pipeline as Code = la configuration du pipeline est stockée sous forme de code (YAML, Groovy, etc.) dans le dépôt Git, ce qui permet versioning, revue de code et traçabilité.

</details>

### Q5 — Artefact dans un pipeline CI/CD

**Question :**
Dans un pipeline moderne, on parle souvent d’“artefact” (build artifact). Lequel des éléments suivants est un **exemple correct d’artefact** ?

* [ ] A. Un message Slack indiquant que les tests ont échoué.
* [ ] B. Une image Docker versionnée poussée dans un registry (ex : `backend-api:1.3.7`).
* [ ] C. Une note manuscrite décrivant le déploiement.
* [ ] D. Un jar/war généré par Maven et stocké dans un repository d’artefacts.

<details>
  <summary>Réponse :</summary>

Bonnes réponses : **B** et **D**.
Un artefact est un résultat **binaire ou packagé** du build (image Docker, jar/war, package nuget, etc.), servant ensuite au déploiement.

</details>

* [ ] B. Rolling Update
* [ ] C. Déploiement “Russian Roulette” (on déploie au hasard sur un serveur)

### Q8 — Tests et CI

**Question :**
Dans le cadre de l’intégration continue, quels types de tests sont typiquement exécutés à chaque commit (ou presque) ?

* [ ] A. Tests unitaires
* [ ] B. Tests d’intégration légers
* [ ] C. Tests end-to-end très lourds de plusieurs heures
* [ ] D. Lint / analyse statique de base (style, règles simples)

<details>
  <summary>Réponse :</summary>

Bonnes réponses : **A**, **B** et **D**.
À chaque commit, on privilégie les tests rapides : unitaires, quelques tests d’intégration ciblés, lint. Les tests très lourds (end-to-end, performance) sont souvent déclenchés moins fréquemment (la nuit, avant release, etc.).

</details>

### Q11 — Anti-pattern CI/CD

**Question :**
Laquelle des situations suivantes est un **anti-pattern** CI/CD évident ?

* [ ] A. Le pipeline échoue parfois, l’équipe analyse les causes, corrige et renforce les tests.
* [ ] B. Les déploiements en production se font en copiant manuellement des fichiers via FTP sur les serveurs, sans logs ni scripts.
* [ ] C. Le pipeline est défini dans un Jenkinsfile versionné dans Git.
* [ ] D. En cas d’incident, l’équipe fait un post-mortem et ajuste le pipeline.

<details>
  <summary>Réponse :</summary>

Bonne réponse : **B**.
Le déploiement manuel par FTP, sans script, sans logs, est un anti-pattern classique, opposé à l’esprit CI/CD (automatisation, traçabilité, reproductibilité).

</details>

### Q12 — Question de synthèse (ouverte)

**Question :**
Expliquez (en 4–5 phrases) comment CI, Continuous Delivery et Continuous Deployment s’enchaînent dans un pipeline moderne, depuis le commit d’un développeur jusqu’à la mise en production.

<details>
  <summary>Réponse :</summary>

Réponse possible (exemple) :
Un développeur pousse un commit sur la branche principale d’un dépôt Git. Cela déclenche la **CI** : le serveur CI récupère le code, exécute le build et les tests automatiques, et produit un artefact (par exemple une image Docker) si tout est vert. En **Continuous Delivery**, cet artefact est ensuite automatiquement déployé sur des environnements de test/staging, avec des tests complémentaires, de sorte que l’application soit toujours prête pour la production. En **Continuous Deployment**, si toutes les étapes réussissent, la mise en production est déclenchée automatiquement, sans intervention humaine, éventuellement avec une stratégie progressive (canary, blue/green). Dans tous les cas, le système est surveillé (logs, métriques, alertes), et on peut rollback si nécessaire.

</details>
